# 大阪府マンション㎡単価インタラクティブWebサイト 設計書

- 作成日: 2026-05-18
- ステータス: ドラフト（ユーザレビュー待ち）

## 1. 目的

国土交通省「不動産情報ライブラリ」API（XIT001）から取得した、大阪府72市区町村のマンション成約データをもとに、過去5年〜最新の㎡単価推移をブラウザ上でインタラクティブに確認できるWebサイトを構築する。GitHubに公開しGitHub Pagesでホストする。

## 2. スコープ

### 含めるもの
- 大阪府全72市区町村のデータ取得
- 期間: 2021年Q1〜APIから取得可能な最新四半期（推定 2025年Q4 or 2026年Q1）
- 集計軸: 市区町村 × 築年数（5区分） × 専有面積（6区分）
- 指標: ㎡単価平均、サンプル件数
- インタラクティブな時系列推移グラフ、市区町村比較、ヒートマップ
- データ取得スクリプトのリファクタ
- GitHub公開（Pages有効化）

### 含めないもの
- 大阪府以外の都道府県
- 戸建てや土地など、マンション以外の物件種別
- ユーザ認証、コメント機能
- データの自動更新（GitHub Actions）。将来の拡張余地として設計はするが、初版は手動実行

## 3. アーキテクチャ概要

```
[Pythonスクリプト]                  [静的Webサイト]
fetch_data.py                       docs/
   ↓ API呼び出し                      ├─ index.html
   ↓ 集計                              ├─ app.js
   ↓ JSON出力                          ├─ style.css
docs/data.json  ──────────────→     └─ data.json （fetchで読み込み）
```

- データ取得（手動・ローカル実行）→ JSON出力 → git push → GitHub Pages配信
- フロントエンドはビルド不要の素のHTML/JS。Plotly.jsはCDN読み込み

## 4. コンポーネント設計

### 4.1 データ取得層（Python）

ファイル構成:
```
scripts/
├─ fetch_data.py        # メインスクリプト
├─ config.py            # 設定（市区町村リスト、定数）
└─ secrets.py           # APIキー（.gitignoreで除外）
secrets.example.py      # APIキーのテンプレ（git管理）
```

責務:
- `secrets.py` からAPIキー読み込み
- 市区町村 × 年 × 四半期のループでAPI呼び出し
- レスポンスをフィルタ（マンション成約価格情報のみ）
- 築年数・面積帯ビンに集計、㎡単価平均と件数を算出
- `docs/data.json` に出力

集計関数の責務分離:
- `fetch_quarter(year, quarter, city_code)`: 1回のAPI呼び出し
- `aggregate(records, age_bins, area_bins)`: フィルタ＆集計
- `build_dataset(years_quarters, cities)`: 全体組み立て
- 各関数は独立してテスト可能

レート制御:
- リクエスト間 0.3秒スリープ
- 失敗時は最大3回リトライ（指数バックオフ）
- 既に取得済みの (city, year, quarter) はキャッシュファイルでスキップ可能

### 4.2 JSONデータ仕様

`docs/data.json`:
```json

    "year_quarter_range": ["2021Q1", "2026Q1"],

    "area_bins": [[50,59],[60,69],[70,79],[80,89],[90,99],[100,120]]
  },
  "cities": [
    {"code": "27102", "name": "都島区"},
    ...
  ],
  "data": {
    "27102": {
      "2021Q1": {
        "0-9": {
          "50-59": {"avg": 850000, "n": 12},
          ...
        },
        ...
      },
      ...
    },
    ...
  }

```

サイズ見積もり:
- 72市区町村 × 20四半期 × 5築年数区分 × 6面積帯 = 43,200セル
- 1セルあたり約30バイト → 約1.3MB（gzip後 数百KB）
- Plotly.jsで十分に処理可能

### 4.3 フロントエンド

`docs/index.html`: シングルページ。コントロール部とグラフ部を縦に並べる。
`docs/app.js`: data.jsonをfetchしてPlotly.jsで描画。フレームワーク不使用。
`docs/style.css`: 最小限のスタイル。レスポンシブ対応（モバイルで縦並びになる）

#### グラフ仕様

**グラフ1: 時系列推移**
- ライブラリ: Plotly.js (`scatter`, mode: 'lines+markers')
- データ: 選択した市区町村 × 築年数 × 面積帯の、四半期ごとの㎡単価
- ホバー時: ㎡単価、件数を表示
- サンプル件数5未満の点は半透明・薄い色で警告

**グラフ2: 市区町村比較**
- ライブラリ: Plotly.js (`bar`)
- データ: 選択した築年数・面積帯で、全72市区町村の最新四半期の㎡単価
- 値順ソートトグル / 五十音順
- データなしの市区町村は除外

**グラフ3: ヒートマップ**
- ライブラリ: Plotly.js (`heatmap`)
- データ: 選択した市区町村の最新四半期について、築年数（X軸）× 面積帯（Y軸）の㎡単価
- カラースケール: Viridis
- セル内に値表示

## 5. データフロー

1. 開発者がローカルで `python scripts/fetch_data.py` を実行
2. スクリプトがAPIから2021Q1〜最新までを順次取得（5〜10分程度）
3. 集計結果が `docs/data.json` として書き出される
4. `git add docs/data.json && git commit && git push`
5. GitHub Pagesが自動デプロイ
6. ユーザがサイトにアクセス → `data.json` をfetch → Plotlyで描画

## 6. エラーハンドリング

- **API失敗**: 最大3回リトライ後、該当四半期はスキップしログ出力
- **データ無し**: JSON上は欠損（キーが存在しない）として表現。フロントエンドは「データなし」を明示
- **サンプル件数が極端に少ない**: グラフ上で警告表示（5件未満を目安）
- **`data.json` の読み込み失敗**: フロントエンドにエラーメッセージ表示
- **APIキー未設定**: スクリプト起動時に `secrets.py` を確認し、未存在ならテンプレを案内

## 7. テスト方針

- 集計関数（pure function）を中心に単体テスト
  - 既知の入力レコードに対する正しい平均算出
  - 築年数・面積帯のビンマッチング
  - 空データ・欠損データの扱い
- API呼び出し部分はモックでテスト
- 全市区町村取得は手動実行で確認
- フロントは目視確認（ローカルでHTTPサーバ立ち上げ）

## 8. セキュリティ・公開時の注意

- **APIキー**: コードに直書きしない。`secrets.py` を `.gitignore` で除外
- 既存ファイルにAPIキーが書かれているため、既存3ファイルは新リポジトリには含めないか、APIキー部分を削除してからコミット
- 国土交通省APIの利用規約遵守（出典明記）。README・サイトフッターに「データ出典: 国土交通省 不動産情報ライブラリ」を明記

## 9. 既存コードからの主な改良点

| 既存の問題 | 改良 |
|---|---|
| APIキー直書き | 別ファイル化・gitignore |
| 設定の重複（85-92行と189-192行） | 1箇所に統合 |
| 現在年が `2025` ハードコード | `datetime.now().year` から動的 |
| エラー時にスキップだけ、ログ薄い | リトライ＋構造化ログ |
| 出力がExcelのみ・市区町村ごとシート分離 | 単一JSONに統合（フロント用） |
| 全件処理に時間がかかるとき再開不可 | キャッシュで再開可能 |
| マンション以外も混入する可能性 | 種別フィルタ強化（中古マンション等） |

## 10. 既知の不確実性・確認事項

- **API取得可能な最新四半期**: 実行時に確認。通常6ヶ月のタイムラグあり
- **APIのレート制限詳細**: 公式ドキュメントを確認し、必要なら間隔を調整
- **既存スクリプトの `?key=` クエリパラメータ方式**: 駅版コードでは `Ocp-Apim-Subscription-Key` ヘッダ方式を使用。現行APIではヘッダ方式が正しい可能性が高く、要検証
- **GitHubユーザ名・最終的なリポジトリ名**: push直前にユーザに確認

## 11. 開発フェーズ

1. **データ取得スクリプトのリファクタ**: 関数分離、設定外出し、リトライ、JSON出力

3. **全件取得**: 72市区町村×20四半期で `data.json` 生成
4. **フロントエンド**: HTML/CSS/JS + Plotly.jsで3グラフ実装
5. **GitHub公開**: 新規リポジトリ作成、Pages有効化

各フェーズの終わりにユーザに確認を取りながら進める。
